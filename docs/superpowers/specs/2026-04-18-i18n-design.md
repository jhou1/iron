# i18n Design: English + Chinese (Simplified)

## Overview

Add multi-language support to ironcli using Project Fluent. The app auto-detects the user's system locale and displays all UI and CLI text in English or Chinese (Simplified).

## Decisions

| Decision | Choice |
|---|---|
| Languages | English + Chinese (Simplified) |
| Locale detection | Auto-detect from `LANG`/`LC_ALL` env vars |
| Translation framework | Project Fluent (`fluent-bundle` + `unic-langid`) |
| String embedding | Compile-time via `include_str!` |
| Units | Fully translated (`kg` → `公斤`, `km` → `公里`, `min` → `分钟`) |
| Date format | Keep ISO `YYYY-MM-DD` in both languages |
| Keybindings | Keys unchanged, labels translated (`[l] Log` → `[l] 记录`) |
| CLI text | Translated (help text, export/import messages) |
| Fallback | Missing Chinese key falls back to English |

## Architecture

### New Files

```
locales/
  en.ftl          — English translations (~100 keys)
  zh-CN.ftl       — Chinese translations (~100 keys)
src/
  i18n.rs         — locale detection, bundle init, tr()/tr_args()
```

### Dependencies

Add to `Cargo.toml`:
- `fluent-bundle` — Fluent message resolution
- `unic-langid` — language identifier parsing

### i18n Module (`src/i18n.rs`)

Public API:

```rust
pub fn init();
pub fn tr(key: &str) -> String;
pub fn tr_args(key: &str, args: &[(&str, FluentValue)]) -> String;
```

- `init()` reads `LANG`/`LC_ALL`, determines locale, builds a `FluentBundle` with the appropriate `.ftl` file, and stores it in a `thread_local!` static.
- `tr()` looks up a simple key and returns the resolved string.
- `tr_args()` looks up a key with named arguments for interpolation.
- If the detected locale starts with `zh`, use `zh-CN.ftl`; otherwise use `en.ftl`.
- Fallback: if a key is missing from the selected locale's `.ftl` file, resolve from `en.ftl`.

### Initialization

`main.rs` calls `i18n::init()` before dispatching to TUI or CLI commands. The bundle is stored in `thread_local!` storage — safe because the app is single-threaded.

## Call Site Changes

All TUI screen files and CLI output code change from inline literals to `tr()`/`tr_args()` calls.

**Simple strings:**
```rust
// Before
Span::styled("Dashboard", Style::default().fg(ACCENT))
// After
Span::styled(tr("dashboard-title"), Style::default().fg(ACCENT))
```

**Interpolated strings:**
```rust
// Before
format!("{} sessions", self.stats.sessions)
// After
tr_args("dashboard-sessions", &[("count", self.stats.sessions.into())])
```

**Practice type labels:**
```rust
// Before: PracticeType::label() returns hardcoded &str
// After: PracticeType::label() calls tr() with the appropriate key
```

### What Changes

- Each screen file in `src/tui/` gets `use crate::i18n::{tr, tr_args};`
- All inline string literals replaced with `tr()` calls
- All `format!()` with user-facing text replaced with `tr_args()` calls
- `PracticeType::label()` routes through `tr()`
- `main.rs` calls `i18n::init()` at startup
- `main.rs` CLI output uses `tr()`/`tr_args()`
- clap `about` attributes use `tr()` for CLI help text

### What Stays Unchanged

- All keybindings (actual keys `j/k/h/l/q` etc.)
- Date formatting (`YYYY-MM-DD`)
- Database schema and stored values
- Export/import JSON format
- Widget implementations (heatmap, sparkline rendering logic)
- Data model (`model.rs` types)
- Database queries (`db.rs`)

## Testing

### i18n Unit Tests

- `init()` works for both `en` and `zh` locales
- `tr()` returns correct strings per locale
- `tr_args()` interpolates values correctly in both locales
- Fallback: missing key in `zh-CN.ftl` returns the English string

### Translation Completeness Test

A test that parses both `.ftl` files and asserts they have the same set of keys. This catches missing translations at test time.

### Existing Tests

The 23 existing tests (model, db, export) do not touch UI strings and require no changes.
